Phase B.1

MariaDB Galera

Cluster Galera 3 nœuds en synchronous multi-master. Anti-affinité HARD : 1 pod par VM. SST via mariabackup.

Secret

YAMLmariadb-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: mariadb-passwords
  namespace: wordpress
type: Opaque
stringData:
  mariadb-root-password: "root_P@ss123"
  mariadb-password:      "user_P@ss123"
  mariadb-galera-mariabackup-password: "maria_Repli123"

Values

YAMLmariadb-values.yaml
image:
  repository: bitnamilegacy/mariadb-galera
  tag: 11.4.5-debian-12-r1

existingSecret: "mariadb-passwords"

db:
  user: wordpress
  name: wordpress

# ── HA ──────────────────────────────────────────────────────
replicaCount: 3
podAntiAffinityPreset: hard  # 1 pod/VM obligatoire

galera:
  mariabackup:
    existingSecret: "mariadb-passwords"

# ── Stockage ────────────────────────────────────────────────
persistence:
  enabled: true
  storageClass: "longhorn"
  size: 1Gi

# ── Ressources ──────────────────────────────────────────────
resources:
  requests:
    cpu: 250m
    memory: 512Mi
  limits:
    cpu: 1000m
    memory: 1500Mi

Déploiement

BASH
kubectl apply -f mariadb-secret.yaml

helm install mariadb-galera oci://registry-1.docker.io/bitnamicharts/mariadb-galera \
  --namespace wordpress \
  -f mariadb-values.yaml \
  --wait --timeout 10m

Validation

BASH
# Les 3 pods doivent être Running
kubectl get pods -n wordpress -l app.kubernetes.io/name=mariadb-galera

# Test de connexion depuis un pod
kubectl exec -n wordpress -it mariadb-galera-0 -- \
  mysql -u wordpress -puser_P@ss123 -e "SHOW STATUS LIKE 'wsrep_cluster_size';"
# Attendu : wsrep_cluster_size = 3
⚠️
Au premier démarrage, le pod mariadb-galera-0 bootstrap le cluster. Les pods 1 et 2 attendent leur IST/SST — c'est normal si le démarrage prend 2-3 minutes.
Phase B.2

Redis Sentinel

Architecture replication + Sentinel. 3 nœuds pour le quorum (quorum: 2). masterSet nommé redis-master.

🔴
Bug dans ton redis-values.yaml d'origine : Le bloc master: est dupliqué — le second (extraFlags) écrase le premier (persistence). Le fichier corrigé ci-dessous fusionne les deux.

Secret

YAMLredis-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: redis-passwords
  namespace: wordpress
type: Opaque
stringData:
  redis-password:          "R€disP@ss123"
  redis-sentinel-password: "5entinel_P@ss123"

Values (corrigé — blocs master/replica fusionnés)

YAMLredis-values.yaml
image:
  repository: bitnamilegacy/redis
  tag: 7.2.4-debian-12-r11

# ── Mode HA ─────────────────────────────────────────────────
architecture: replication
replicaCount: 3

# ── Auth nœuds Redis ────────────────────────────────────────
auth:
  existingSecret: "redis-passwords"
  existingSecretPasswordKey: "redis-password"

# ── Sentinel ────────────────────────────────────────────────
sentinel:
  enabled: true
  masterSet: redis-master   # ⚠️ WordPress devra utiliser exactement ce nom
  quorum: 2
  image:
    repository: bitnamilegacy/redis-sentinel
    tag: 7.2.4-debian-12-r11
  auth:
    existingSecret: "redis-passwords"
    existingSecretPasswordKey: "redis-sentinel-password"

# ── Anti-affinité ───────────────────────────────────────────
podAntiAffinityPreset: hard

# ── Master (bloc UNIQUE — persistence + extraFlags fusionnés) 
master:
  persistence:
    enabled: true
    storageClass: "longhorn"
    size: 2Gi
  extraFlags: []  # désactive le chargement de modules inexistants

# ── Replica ─────────────────────────────────────────────────
replica:
  persistence:
    enabled: true
    storageClass: "longhorn"
    size: 2Gi
  extraFlags: []

commonConfiguration: |
  # Config redis minimale — évite le chargement de modules

Déploiement

BASH
kubectl apply -f redis-secret.yaml

helm install redis-ha oci://registry-1.docker.io/bitnamicharts/redis \
  --namespace wordpress \
  -f redis-values.yaml \
  --wait --timeout 5m

Validation

BASH
# Les 3 pods doivent être Running (chacun contient : redis + sentinel)
kubectl get pods -n wordpress -l app.kubernetes.io/name=redis

# Vérifier que Sentinel connaît le master
kubectl exec -n wordpress redis-ha-node-0 -c redis -- \
  redis-cli -p 26379 -a '5entinel_P@ss123' \
  SENTINEL masters
# Cherche la ligne "name" → redis-master

# Test ping Redis via Sentinel
kubectl exec -n wordpress redis-ha-node-0 -c redis -- \
  redis-cli -a 'R€disP@ss123' PING
# Attendu : PONG
Phase C

MinIO Distributed

Mode distribué 4 pods avec Erasure Coding (EC 2+2). Anti-affinité SOFT obligatoire (4 pods / 3 nœuds). Interface console intégrée sur NodePort 30091.

Values

YAMLminio-values.yaml
image:
  repository: bitnamilegacy/minio
  tag: 2024.11.7-debian-12-r0

# ── Mode distribué ──────────────────────────────────────────
mode: distributed
replicas: 4

# ── Auth ────────────────────────────────────────────────────
auth:
  rootUser:     "admin-ha"
  rootPassword: "mini0_1234"

# ── Console ─────────────────────────────────────────────────
console:
  enabled: false

api:
  browser: "on"

# ── Persistence ─────────────────────────────────────────────
persistence:
  enabled: true
  storageClass: "longhorn"
  size: 5Gi

# ── Anti-affinité SOFT (4 pods / 3 nœuds) ───────────────────
podAntiAffinityPreset: soft

# ── Service NodePort ────────────────────────────────────────
service:
  type: NodePort
  ports:
    api: 9000
    console: 9001
  nodePorts:
    api: 30090
    console: 30091

# ── URLs console & API ───────────────────────────────────────
extraEnvVars:
  - name: MINIO_OPTS
    value: "--console-address :9001"
  - name: MINIO_SERVER_URL
    value: "http://192.168.90.10:30090"
  - name: MINIO_BROWSER_REDIRECT_URL
    value: "http://192.168.90.10:30091"

# ── Ressources ──────────────────────────────────────────────
resources:
  requests:
    memory: 256Mi
    cpu: 100m
  limits:
    memory: 512Mi
    cpu: 500m

Déploiement

BASH
helm install minio-ha oci://registry-1.docker.io/bitnamicharts/minio \
  --namespace wordpress \
  -f minio-values.yaml \
  --wait --timeout 10m

Tests MinIO HA

ℹ️
Récupère les credentials avant de lancer le pod client.
BASH
# Exporter les variables
export ROOT_USER=admin-ha
export ROOT_PASSWORD=mini0_1234

# Lancer le pod client mc
kubectl run --namespace wordpress minio-ha-client \
  --rm --tty -i --restart='Never' \
  --env MINIO_SERVER_ROOT_USER=$ROOT_USER \
  --env MINIO_SERVER_ROOT_PASSWORD=$ROOT_PASSWORD \
  --env MINIO_SERVER_HOST=minio-ha \

# Configurer l'alias
mc alias set myminio http://$MINIO_SERVER_HOST:9000 \
  $MINIO_SERVER_ROOT_USER $MINIO_SERVER_ROOT_PASSWORD

# ── Vérification santé cluster ──────────────────────────────
mc admin info myminio
# Cherche : "4 drives online, 0 drives offline"

# ── Créer le bucket WordPress ───────────────────────────────

mc anonymous set download myminio/wordpress-media

# ── Créer un compte dédié WordPress (ne pas utiliser root) ──

mc admin policy attach myminio readwrite --user wp-user

# ── Tests fonctionnels ──────────────────────────────────────
mc mb myminio/test-bucket
echo "minio test data" > test.txt && mc cp test.txt myminio/test-bucket/
mc ls myminio/test-bucket/
mc rm myminio/test-bucket/test.txt
mc rb myminio/test-bucket

Phase E

Plugins & activation WP-CLI

⚠️
Pourquoi WP-CLI et pas l'interface web ? Les plugins Redis Object Cache et WP Offload Media installés via l'UI ne lisent pas wp-config.php pour leur configuration avancée (Sentinel, force-path-style). Sans WP-CLI, ils tentent une connexion TCP directe sur le master Redis — qui peut changer d'IP après un failover — et échouent avec MinIO sur le virtual-hosted style.

Installation et activation (une seule fois)

BASH
# Attendre que les pods soient Ready
kubectl wait --for=condition=ready pod \
  -l app.kubernetes.io/name=wordpress \
  -n wordpress --timeout=300s

# Exec dans le pod
kubectl exec -n wordpress -it deploy/wordpress -- bash
BASHdans le pod WordPress
# ── Plugin Redis Object Cache ───────────────────────────────
wp plugin install redis-cache --activate --allow-root
wp redis enable --allow-root
wp redis status --allow-root
# Attendu : Status: Connected  |  Client: Predis  |  Sentinel: redis-master

# ── Plugin WP Offload Media Lite ────────────────────────────
wp plugin install amazon-s3-and-cloudfront --activate --allow-root

# Vérifier que AS3CF_SETTINGS est bien lu depuis wp-config
wp eval "echo AS3CF_SETTINGS;" --allow-root
# Doit retourner la chaîne sérialisée (non vide)

# ── Test upload vers MinIO ───────────────────────────────────
wp media import https://via.placeholder.com/300.png --allow-root
# L'image doit apparaître dans le bucket wordpress-media
✅
Les plugins sont installés dans le PV RWX Longhorn — ils sont partagés entre tous les pods WordPress. Aucune réinstallation nécessaire lors d'un scale ou d'un restart.
Phase F

Scaling & validation

Scale horizontal WordPress

BASH
# Scale à 4 replicas
helm upgrade wordpress oci://registry-1.docker.io/bitnamicharts/wordpress \
  --namespace wordpress \
  --reuse-values \
  --set replicaCount=4

# Vérifier que tous les pods montent le même PVC RWX
kubectl get pods -n wordpress -l app.kubernetes.io/name=wordpress \
  -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.volumes[?(@.name=="wordpress-data")].persistentVolumeClaim.claimName}{"\n"}{end}'

Test de failover Redis Sentinel

BASH
# Identifier le master actuel
kubectl exec -n wordpress redis-ha-node-0 -c redis -- \
  redis-cli -p 26379 -a '5entinel_P@ss123' SENTINEL get-master-addr-by-name redis-master

# Tuer le pod master
kubectl delete pod redis-ha-node-0 -n wordpress

# Observer le failover (Sentinel élit un nouveau master en ~10s)
watch kubectl get pods -n wordpress -l app.kubernetes.io/name=redis

# Vérifier que WordPress continue de fonctionner
curl -s -o /dev/null -w "%{http_code}" http://wordpress.local
Phase G

Troubleshooting

SymptômeCause probableCommande diagnostic
Redis : Not connected masterSet incorrect ou mot de passe Sentinel manquant wp redis status --allow-root + vérifier WP_REDIS_SENTINEL_SERVICE
MinIO : 403 / SignatureDoesNotMatch force-path-style absent ou credentials incorrects wp eval "var_dump(AS3CF_SETTINGS);" --allow-root
PVC en Pending Longhorn ne supporte pas RWX sur cette version kubectl describe pvc -n wordpress
Galera : pod crashloop Bootstrap séquentiel — pods 1&2 attendent pod 0 kubectl logs mariadb-galera-0 -n wordpress
WP inaccessible après scale Session PHP non partagée entre pods Activer Redis pour les sessions via Redis Object Cache
